Spring Annotations — Pismo Project
====================================

====================
 APLICAÇÃO
====================

@SpringBootApplication
  Ponto de entrada da aplicação. Combina três anotações em uma:
  - @Configuration: marca a classe como fonte de beans do Spring
  - @EnableAutoConfiguration: ativa a configuração automática baseada nas dependências do classpath
  - @ComponentScan: faz o Spring escanear e registrar automaticamente os componentes do pacote atual e subpacotes
  Usado em: Main.java


====================
 CAMADA WEB (Controller)
====================

@RestController
  Combina @Controller + @ResponseBody.
  Indica que a classe é um controlador REST e que todos os métodos retornam dados diretamente no corpo da resposta HTTP (JSON), sem renderizar views.
  Usado em: ClientAccountController.java, TransactionController.java

@RequestMapping
  Define o prefixo de URL base para todos os endpoints de um controller.
  Exemplo: @RequestMapping("/accounts") faz todos os métodos daquele controller responderem a /accounts.
  Usado em: ClientAccountController.java, TransactionController.java

@GetMapping
  Atalho para @RequestMapping(method = RequestMethod.GET).
  Mapeia requisições HTTP GET para um método específico.
  Usado em: ClientAccountController.java

@PostMapping
  Atalho para @RequestMapping(method = RequestMethod.POST).
  Mapeia requisições HTTP POST para um método específico.
  Usado em: ClientAccountController.java, TransactionController.java

@PathVariable
  Extrai um valor da URL e injeta como parâmetro do método.
  Exemplo: /accounts/{id} → @PathVariable Long id captura o valor de {id}.
  Usado em: ClientAccountController.java

@RequestBody
  Desserializa o corpo da requisição HTTP (JSON) e injeta como parâmetro do método.
  O Spring usa o Jackson para converter o JSON no objeto Java correspondente.
  Usado em: ClientAccountController.java, TransactionController.java

@Valid
  Dispara a validação do Bean Validation (JSR-380) no objeto anotado.
  Sem esta anotação, as constraints (@NotNull, @Pattern, etc.) no DTO são ignoradas.
  Usado em conjunto com @RequestBody.
  Usado em: ClientAccountController.java, TransactionController.java


====================
 TRATAMENTO DE ERROS
====================

@RestControllerAdvice
  Marca a classe como um handler global de exceções para todos os controllers.
  Combina @ControllerAdvice + @ResponseBody, permitindo retornar respostas JSON nos handlers de erro.
  Usado em: GlobalExceptionHandler.java

@ExceptionHandler
  Marca um método para tratar um tipo específico de exceção lançada por qualquer controller.
  O Spring escolhe o handler mais específico primeiro — ex: AccountNotFoundException antes de RuntimeException.
  Exemplo: @ExceptionHandler(AccountNotFoundException.class) intercepta apenas aquela exceção.
  Usado em: GlobalExceptionHandler.java


====================
 EXCEÇÕES CUSTOMIZADAS
====================

Conceito: extends RuntimeException
  Cria uma exceção unchecked com semântica de negócio clara.
  Unchecked significa que o compilador não obriga try/catch — a exceção sobe a pilha até o GlobalExceptionHandler.
  Usar RuntimeException (não Exception) evita poluir os métodos dos services com throws ou try/catch.

  Estrutura base:
    public class MinhaException extends RuntimeException {
        public MinhaException() {
            super("mensagem fixa");          // mensagem fixa: sempre o mesmo motivo
        }
        public MinhaException(String msg) {
            super(msg);                      // mensagem variável: motivo depende do contexto
        }
    }

  Quando usar mensagem fixa vs variável:
    - Fixa: quando a exceção sempre representa o mesmo motivo (ex: AccountNotFoundException → "Account not found")
    - Variável: quando a exceção pode ocorrer por motivos diferentes (ex: InvalidTransactionException → amount inválido, tipo inválido)

Exceções criadas no projeto:
  AccountNotFoundException    → HTTP 404 Not Found       — conta não encontrada pelo id
  DuplicateDocumentException  → HTTP 409 Conflict        — documento já cadastrado
  InvalidTransactionException → HTTP 422 Unprocessable Entity — regra de negócio da transação violada

  Arquivos: com/pismo/exception/AccountNotFoundException.java
            com/pismo/exception/DuplicateDocumentException.java
            com/pismo/exception/InvalidTransactionException.java

Tradeoff de segurança: HTTP status e user enumeration
  - 404 em GET /accounts/{id} pode confirmar que um recurso não existe (user enumeration).
  - 422 em POST /transactions não expõe estado do banco — apenas informa que a regra de negócio não foi satisfeita.
  - Em APIs sem autenticação, avaliar retornar respostas genéricas para evitar vazamento de informação.
  - A mensagem no body é onde o vazamento acontece, não o status HTTP em si.


====================
 CAMADA DE SERVIÇO
====================

@Service
  Marca a classe como um componente de serviço no contexto do Spring.
  É detectado pelo @ComponentScan e registrado como bean, permitindo injeção de dependência.
  Semanticamente indica que a classe contém lógica de negócio.
  Usado em: ClientAccountService.java, TransactionService.java


====================
 CAMADA DE PERSISTÊNCIA (Model / JPA)
====================

@Entity
  Marca a classe como uma entidade JPA, ou seja, ela será mapeada para uma tabela no banco de dados.
  Obrigatória para que o Hibernate reconheça a classe.
  Usado em: ClientAccountModel.java, OperationTypeModel.java, TransactionModel.java

@Table
  Define o nome da tabela no banco de dados que a entidade representa.
  Sem esta anotação, o Hibernate usa o nome da classe como nome da tabela.
  Exemplo: @Table(name = "accounts")
  Usado em: ClientAccountModel.java, OperationTypeModel.java, TransactionModel.java

@Id
  Marca o campo como chave primária da entidade.
  Obrigatório em toda entidade JPA.
  Usado em: ClientAccountModel.java, OperationTypeModel.java, TransactionModel.java

@GeneratedValue
  Define a estratégia de geração automática do valor da chave primária.
  strategy = GenerationType.IDENTITY delega a geração ao banco (ex: SERIAL / AUTO_INCREMENT no PostgreSQL).
  Não usado em OperationTypeModel pois os IDs são fixos e inseridos via data.sql.
  Usado em: ClientAccountModel.java, TransactionModel.java

@Column
  Configura o mapeamento de um campo para uma coluna específica no banco.
  Permite definir: nome da coluna, se é nullable, se é unique, entre outros.
  Exemplo: @Column(name = "document_number", nullable = false, unique = true)
  Usado em: ClientAccountModel.java, OperationTypeModel.java, TransactionModel.java

@PrePersist
  Marca um método para ser executado automaticamente pelo JPA antes de inserir a entidade no banco.
  Usado para popular o campo eventDate com o timestamp atual no momento da criação da transação.
  Usado em: TransactionModel.java


====================
 VALIDAÇÃO (Bean Validation)
====================

@NotNull
  Garante que o campo não seja nulo. Não valida se está vazio.
  Usado em campos de tipo objeto (Long, BigDecimal) no TransactionRequest.
  Usado em: TransactionRequest.java

@Pattern
  Valida que o campo String segue uma expressão regular específica.
  Exemplo: @Pattern(regexp = "\\d{11}") garante exatamente 11 dígitos numéricos.
  Usado em: ClientAccountRequest.java


====================
 TESTES UNITÁRIOS
====================

Conceitos gerais
  Teste unitário testa a lógica de um método de forma isolada, sem subir o Spring, sem banco de dados.
  Testa se um método retorna o resultado correto dado um input simulado (mockado).
  Não testa integração entre camadas nem comportamento end-to-end.

  Cada caminho de código relevante merece um teste:
    - Caminho de sucesso (happy path)
    - Cada caminho de erro (exceções esperadas)

@ExtendWith(MockitoExtension.class)
  Habilita o Mockito no JUnit 5, permitindo o uso de @Mock e @InjectMocks.
  Sem esta anotação, os mocks não são inicializados automaticamente.
  Usado em: ClientAccountServiceTest.java, TransactionTest.java

@Mock
  Cria um "dublê" do objeto — substitui a dependência real por uma versão controlada.
  O mock não executa lógica real; você programa o comportamento com when(...).thenReturn(...).
  Usado para simular repositories sem tocar no banco de dados.
  Usado em: ClientAccountServiceTest.java, TransactionTest.java

@InjectMocks
  Cria uma instância real da classe sendo testada e injeta os @Mocks como dependências.
  A classe testada não sabe que está recebendo mocks — acha que são as dependências reais.
  Usado em: ClientAccountServiceTest.java, TransactionTest.java

Mockando comportamentos
  when(repo.findById(1L)).thenReturn(Optional.of(obj))  → simula objeto encontrado
  when(repo.findById(99L)).thenReturn(Optional.empty()) → simula objeto não encontrado
  when(repo.existsById(1L)).thenReturn(true)            → simula existência
  when(repo.save(any(Classe.class))).thenReturn(obj)    → simula persistência

  any(Classe.class): matcher do Mockito que aceita qualquer instância daquela classe como argumento.

Assertions principais (JUnit 5)
  assertNotNull(obj)                          → garante que o objeto não é nulo
  assertEquals(esperado, real)               → garante igualdade de valor
  assertFalse(condicao)                       → garante que a condição é falsa
  assertThrows(Excecao.class, () -> { ... }) → garante que o bloco lança aquela exceção
    - Retorna a exceção lançada, permitindo verificar a mensagem com getMessage()
    - Usar a classe específica (ex: AccountNotFoundException.class), não RuntimeException.class
      Razão: ser preciso evita que o teste passe por uma exceção errada

Optional nos testes
  findById retorna Optional<T> porque o registro pode não existir.
  Optional.of(obj)    → container com valor presente
  Optional.empty()    → container vazio, simula "não encontrado"
  O service usa .orElseThrow() para lançar a exceção quando o Optional está vazio.

Testando listas
  when(repo.findByAccountId(1L)).thenReturn(List.of())           → simula lista vazia
  when(repo.findByAccountId(1L)).thenReturn(List.of(obj1, obj2)) → simula lista com itens
  List.of() cria uma lista vazia imutável — jeito idiomático em Java moderno.

  Assertions para listas:
    assertTrue(response.isEmpty())    → verifica que a lista está vazia
    assertEquals(2, response.size())  → verifica quantidade de itens
    assertNotNull(response)           → verifica que a lista não é nula

  Três casos a testar em métodos que retornam listas:
    1. Recurso pai não existe → exceção
    2. Recurso pai existe, sem itens → lista vazia
    3. Recurso pai existe, com itens → lista com os itens
  Separar casos 2 e 3 é boa prática: "sem resultados" e "com resultados" são comportamentos distintos.

Spring Data JPA — Query por nome de método
  O Spring Data gera queries automaticamente a partir do nome do método no repository.
  Padrão: findBy + NomeDoCampo
  Exemplo: findByAccountId(Long accountId) → SELECT * FROM transactions WHERE account_id = ?
  Retorna List<T> quando pode haver múltiplos resultados, Optional<T> quando é único.
  Usado em: TransactionRepo.java


====================
 TRANSAÇÕES DE BANCO
====================

@Transactional
  Garante atomicidade na operação — tudo ou nada.
  Se qualquer operação dentro do método falhar, o Spring faz rollback automático de tudo que foi feito no banco.
  Gerenciado pelo Spring via proxy: o Spring intercepta a chamada ao método e envolve com abertura/commit/rollback de transação.

  Onde usar:
    - Métodos de escrita: save, update, delete → @Transactional
    - Métodos de leitura: find, get → não precisa (opcional: @Transactional(readOnly = true) para otimização)

  Onde NÃO colocar:
    - Na classe inteira, a menos que todos os métodos sejam de escrita
    - Em métodos de leitura simples sem necessidade de consistência transacional

  @Transactional(readOnly = true)
    Avisa o banco que não haverá escrita, permitindo otimizações internas.
    Boa prática para métodos de leitura em projetos maiores.

  Usado em: ClientAccountService.createAccount, TransactionService.createTransaction
